The problem

More agents, more chaos.

A second coding agent doesn't double your output — it doubles your coordination overhead. The failure modes are predictable.

They collide on the same file

Two agents open src/auth/login.py at the same moment. One overwrites the other; work is silently lost, or you spend the afternoon untangling a merge. Nothing in a plain repo stops it — coordination is left to luck.

This is the failure that scales worst as you add agents: collisions grow with every parallel worker.

They forget

Every session starts at zero. The same gotcha, re-learned every time.

They're siloed

Switch vendor, lose the context. No shared memory across tools.

Proof, measured honestly

A team that actually gets better.

Not a claim — a reproducible benchmark in the repo. Two results we can stand behind.

−88%
Repeat errors, gone

Polis records each failure as a lesson or guardrail and auto-injects it into matching future tasks, so the repeat-error rate falls from 65% to 8% versus a memoryless setup. Each failure class recurs at most once. A lone agent can't draw that curve — it never accumulates.

# reproduce it yourself
$ polis bench --mode learning
Zero
Collisions, guaranteed

Before touching code, an agent reserves the files. An overlapping reservation is deterministically rejected — no model judgement, no race. The guarantee a single agent or an unmanaged swarm structurally cannot give.

$ polis reserve src/auth --as claude
$ polis reserve src/auth/login.py --as codex
REJECTED — held by claude
How it works

One _polis/ folder. In your repo.

Plain markdown plus two small Python scripts. No server, no database, no lock-in — it lives in git, and every vendor's agent reads the same source of truth.

Step 01
Found a polis

Scaffold the workspace and register your agents as citizens.

$ uvx polis-protocol init
Step 02
Open & route work

Contracts get an owner, acceptance criteria, and a transparent routing reason.

$ polis contract open --title …
$ polis route --explain
Step 03
Settle & compound

Settled work files a lesson; the next matching task starts pre-loaded with it.

$ polis contract settle … ok
Intellectual honesty

When to use it — and when not to.

We benchmarked our own claims. Here's the straight version.

Reach for Polis when
  • 2+ agents touch one real repo and collisions are a risk
  • You switch between vendors and lose context each time
  • You want the team to stop repeating mistakes — provably
  • You need an auditable record of who did what, and why
Skip it when
  • One well-prompted agent already does the job
  • Your capability cards are accurate and tasks never drift
  • You want a hosted runtime — Polis coordinates, it doesn't execute
Install

Start in under five minutes.

# one command, nothing to install
$ uvx polis-protocol init

# or as a persistent CLI
$ pipx install polis-protocol
$ polis status ok
